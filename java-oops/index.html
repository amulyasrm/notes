Part 1: The Core Framework

1. OOP Foundation

What is OOP? Object-Oriented Programming is a paradigm that models software around real-world objects rather than arbitrary logic trees.

  • Why use it? It maps real-world behavior directly to code (e.g., modeling a Bank Account or a Car).
  • Procedural vs OOP: Procedural executes top-down sequence functions globally (C). OOP secures state and behavior into encapsulated instances (Java, Python).
  • Features (The 4 Pillars): Encapsulation, Inheritance, Polymorphism, Abstraction.

2. Classes & Objects

  • Class: A logical blueprint or template (e.g., the architectural drawing of a house).
  • Object: A physical instance of the class consuming exact memory (e.g., the actual house built from the drawing).
  • Creation: Use the `new` keyword to initialize a fresh object. Car bmw = new Car();
  • Instance vs Local: Instance variables belong uniquely to the object throughout its lifetime. Local variables exist purely inside a single method scope and die immediately.
  • `this` keyword: Represents the current executing object dynamically. Used to distinguish instance variables from identically-named local method parameters.

3. The Memory Concept (Stack vs Heap)

  • Heap Memory: Every time you use the `new` keyword, the actual physical Object is allocated inside the vast Heap map.
  • Stack Memory: Holds short-lived local variables and the actual Reference pointers tracking the Heap objects.
  • Scenario: Student s = new Student(); -> The physical student exists on the Heap. The letter `s` lives in the Stack pointing to it.

Part 2: Actions & Scope Modifiers

4. Methods

A method defines the exactly executable behavior of an object.

  • Parameters vs Arguments: Parameters are the variables declared in the method definition. Arguments are the actual values passed during the calling.
  • Static vs Non-Static: Static belongs to the Class entirely (called via `ClassName.method()`). Non-static belongs uniquely to an explicit Object snapshot.
  • Overloading: Creating multiple methods utilizing the exact same name, but demanding entirely different parameter lists conceptually.

5. Constructors

A specialized block to perfectly initialize an object dynamically the moment it hits the Heap.

  • Rules: Must explicitly share the identical class name. Cannot inherently possess a return type (not even `void`).
  • Default vs Parameterized: A Default constructor takes 0 arguments. A Parameterized passes specific initial data.
  • this() Chaining: Permits one constructor to actively call a secondary constructor inside the identical class to reduce code duplication.

Part 3: The 4 Pillars of OOP

Pillar 1: Encapsulation

Definition: Binding data (variables) and logic (methods) securely into a single unit (Data Hiding).

The Mechanics: Private Variables + Public Getters/Setters
  • Private variables: Prevent external classes from directly editing data maliciously.
  • Getter Method: A read-only tunnel to extract the value. public String getName() { return name; }
  • Setter Method: A write tunnel to change the value securely. Provides strict Data Validation internally. public void setAge(int a) { if(a>0) age =a; }
Constructors vs Getters/Setters (Difference & Usage)
  • Constructor: Executes strictly once during object creation for initial birth mapping.
  • Setters: Executes unlimited times specifically for post-birth modifications.
  • Immutable Classes: To create an object that can never be modified (like `String`), solely use a Constructor + Getters. Delete all Setters to freeze it completely.

Pillar 2: Inheritance

Definition: Utilizing one core Parent class fundamentally to automatically supply properties and methods down into Child classes.

  • Concept: It forms a strict IS-A relationship (e.g., A Dog IS-A Animal). Facilitates tremendous Code Reusability.
  • Types Supported: Single, Multilevel, and Hierarchical. Java completely rejects Multiple Inheritance through standard classes forcefully blocking the "Diamond Problem".
  • The `super` Keyword: Points natively exactly at the direct Parent class object. super.method() calls the parent version.

Pillar 3: Polymorphism

Definition: "Many Forms." Permitting identical methods conceptually to execute uniquely differently depending entirely on context.

  • Compile-time (Overloading): Using the identical method name but varying the parameter argument structures. Java natively maps this directly during the compiling phase.
  • Runtime (Method Overriding): When a Child completely rewrites a Parent's method uniquely. The actual decision on which literal method actively triggers resolves dynamically during runtime via "Dynamic Method Dispatch".
Parent p = new Child(); p.draw(); // Calls the Child's draw() exclusively at runtime dynamically!

Part 4: Crucial Architecture Keywords

Contextual Relationships

  • IS-A (Inheritance): A Truck literally IS-A Vehicle structurally. Accomplished smoothly using extends.
  • HAS-A (Composition): A Car HAS-A Engine component dynamically. Accomplished strictly by declaring an Engine variable physically inside the Car class securely.

The Final & Static Concepts

  • Final variable: Completely freezes the value immutably globally. Cannot be altered mathematically.
  • Final method: Absolutely blocks structural Method Overriding universally by child classes completely.
  • Final class: Absolutely blocks structural Inheritance universally. The class completely cannot be extended whatsoever.
  • Static: Means the component is intrinsically bound fundamentally to the Class structurally, completely ignoring individual individual Objects. Accessible purely via standard Class name.

The Object Class

The implicit universal Parent explicitly backing every singular creatively created Java Class globally seamlessly.
