1. Chat Model Fundamentals

Chat Models are the bedrock of LangChain. They represent an abstraction layer that allows developers to interact with various LLMs (GPT, Claude, Gemini) using a unified interface. LangChain v0.2 is the current standard, streamlining these interactions.

Basic Interaction: The `invoke()` Method

The invoke() method is the universal trigger in LangChain. It sends a request to the model and returns a structured response object.

from langchain_openai import ChatOpenAI from dotenv import load_dotenv load_dotenv() # Loads OPENAI_API_KEY from .env model = ChatOpenAI(model="gpt-4o") result = model.invoke("What is 81 divided by 9?") print(result.content) # "9"

Metadata vs. Content: A full result includes metadata (token usage, finish reason, etc.), but result.content extracts only the string response for your application.

2. Prompt Templates

Prompt Templates are the blueprints for LLM inputs. They allow you to decouple the prompt's structure from the specific data being processed, making your code modular and reusable.

The ChatPromptTemplate Powerhouse

In LangChain v0.2/v0.3, ChatPromptTemplate is the standard for building complex message sequences with dynamic placeholders.

from langchain_core.prompts import ChatPromptTemplate from langchain_core.messages import SystemMessage, HumanMessage # Defining a reusable template template = ChatPromptTemplate.from_messages([ ("system", "You are a specialized agent in {topic}."), ("human", "Explain {concept} in {word_count} words."), ]) # Injecting variables prompt_value = template.invoke({ "topic": "Quantum Computing", "concept": "Superposition", "word_count": "50" }) # To see the formatted messages: print(prompt_value.to_messages())

Partial Formatting: You can bind some variables early (e.g., the system persona) and leave others for runtime. This is critical for building template libraries for different specialized agents.

MessagesPlaceholder

For conversational apps, the MessagesPlaceholder is used to dynamically insert the chat history into a template.

from langchain_core.prompts import MessagesPlaceholder template = ChatPromptTemplate.from_messages([ ("system", "You are a helpful assistant."), MessagesPlaceholder(variable_name="history"), ("human", "{input}"), ])

3. Messaging Schema & Logic

Conversations are structured as a List of Messages. To build a dialogue, you must manage different message types:

  • SystemMessage: Sets the behavior/persona and context. Always stays at the top.
  • HumanMessage: Represents the user's input.
  • AIMessage: Represents the model's response.
from langchain_core.messages import SystemMessage, HumanMessage, AIMessage messages = [ SystemMessage(content="You are a professional math tutor."), HumanMessage(content="What is 10 times 5?"), ] result = model.invoke(messages) # To continue the conversation: messages.append(result) # Add the AI response messages.append(HumanMessage(content="Explain how you got that."))

4. Chains & LCEL (LangChain Expression Language)

LCEL is a declarative way to compose chains. It provides a unified interface for streaming, batching, and async support, while allowing you to pipe the output of one component directly into another using the | operator.

The Basic Pipe: Prompt | Model | Parser

The most common chain structure transforms a prompt into a model request, then parses the raw AI output into a clean format.

from langchain_core.output_parsers import StrOutputParser from langchain_core.prompts import ChatPromptTemplate prompt = ChatPromptTemplate.from_template("Tell me a fact about {topic}") model = ChatOpenAI(model="gpt-4o") parser = StrOutputParser() # Creating the chain chain = prompt | model | parser # Execution result = chain.invoke({"topic": "Black Holes"}) print(result) # Directly returns the string content

Standard Interface: Every LCEL chain supports invoke() (sync), ainvoke() (async), stream() (real-time chunks), and batch() (multiple inputs in parallel).

Advanced Composition: Parallel & Passthrough

  • RunnablePassthrough: Used to pass inputs unchanged or combine them with additional context.
  • RunnableParallel: Executes multiple branches simultaneously—perfect for retrieval or multi-model comparisons.
from langchain_core.runnables import RunnableParallel, RunnablePassthrough # Passing input to multiple branches setup_and_retrieval = RunnableParallel( {"context": retriever, "question": RunnablePassthrough()} ) final_chain = setup_and_retrieval | prompt | model | parser

Custom Logic: RunnableLambda

When you need to execute custom Python code that isn't a pre-built LangChain component, use RunnableLambda. It wraps any function into a "Runnable" that can be piped.

from langchain_core.runnables import RunnableLambda # A custom function to clean output def simple_cleaner(text: str): return text.upper().strip() # Wrap and pipe custom_chain = chain | RunnableLambda(simple_cleaner) result = custom_chain.invoke({"topic": "AI"})

6. Enterprise Persistence (Firebase)

In production, you cannot store chat history in local lists—it must be persistent. Using Firebase Firestore allows you to store and retrieve conversation history across sessions.

Persistence Architecture

  • FirestoreChatMessageHistory: Automates the loading and saving of messages to the cloud.
  • Session IDs: Unique identifiers to group messages belonging to a specific user or conversation.
from langchain_google_firestore import FirestoreChatMessageHistory from google.cloud import firestore client = firestore.Client(project="your-project-id") chat_history = FirestoreChatMessageHistory( session_id="user_unique_session_123", collection="chat_history", client=client, ) # Now, any message added is automatically saved to the cloud chat_history.add_user_message("Hello!") chat_history.add_ai_message("How can I help you today?")
Step-by-Step Firebase Setup

1. Create a project at console.firebase.com.

2. Enable Firestore Database in the build tab.

3. Install the GCloud CLI and authenticate using gcloud auth application-default login.

4. Use the project ID and session ID in your LangChain code to start syncing.

1. RAG Foundations

Retrieval-Augmented Generation (RAG) is the paradigm of grounding LLM outputs in verified external data. It solves the "Hallucination" problem by providing the model with a "search engine" for your private documents.

  • Vector Stores: Efficiently storing and retrieving high-dimensional document embeddings (e.g., ChromaDB, Pinecone).
  • Embeddings: Converting text into mathematical vectors that capture semantic meaning.
  • Retrievers: The interface between the model and the vector database, selecting the "Top-K" most relevant chunks.

2. Conversational RAG

Managing History-Aware Retrieval

Standard RAG takes a single query and returns results. Conversational RAG requires the model to understand the *entire* dialogue history to reformulate queries for the vector store.

The Core Loop: Chat History + New Question → Reformulated Query → Vector Search → Grounded Answer.

# History-Aware Retriever Prompt contextualize_q_system_prompt = ( "Given a chat history and the latest user question " "which might reference context in the chat history, " "reformulate a standalone question which can be understood " "without the chat history." )
  • Create Stuff Documents Chain: A LangChain utility that "stuffs" all retrieved documents into a single prompt for the LLM.
  • History-Aware Retriever: Re-phrases the user question based on previous turns to ensure the vector lookup is accurate even with pronouns like "it" or "him".

3. Advanced Web Scraping

Building a RAG system often starts with data collection. LangChain provides multiple loaders, but not all are created equal.

Standard WebLoader

Basic HTML extraction. Often misses dynamic content rendered via JavaScript and includes messy boilerplate (headers, footers).

Firecrawl Loader

Converts websites into clean, LLM-ready Markdown. Handles dynamic JS rendering and provides superior chunking metadata.

Why Firecrawl?

Firecrawl turns the "gross" HTML into a markdown version that is easy for LLMs to navigate. It uses a "crawl" vs "scrape" mode—crawl follows links to map a site, while scrape targets a specific page depth.

# Initializing Firecrawl from langchain_community.document_loaders import FireCrawlLoader loader = FireCrawlLoader( api_key="your_key", url="https://apple.com", mode="scrape" )

4. Autonomous Agents

An Agent is an LLM acting as a State Machine. It doesn't just predict text; it plans, iterates, and executes actions.

The ReAct Loop (Reason + Act)

  • Thought: The agent plans its next move based on the goal and current observation.
  • Action: The agent selects a tool to execute (e.g., "Search Wikipedia").
  • Observation: The agent reads the result of the tool execution and updates its understanding.

Agent Executor: The runtime that manages the loop, handling tool calls and feeding results back into the agent's memory.

5. Tool Development Ecosystem

Tools provide agents with "superpowers"—the ability to interact with the outside world, run code, or access databases.

Types of Tools

  • Tool Constructor: Creating simple tools from Python functions using `@tool` decorators.
  • Structured Tools: Defining complex input schemas using Pydantic models to ensure the agent passes the correct arguments.
  • BaseTool Inheritance: Creating custom classes for maximum control, defining both synchronous and asynchronous behavior.
# Custom Tool with Pydantic Schema class MultiplyNumbersInput(BaseModel): a: float = Field(description="First number") b: float = Field(description="Second number") class MultiplyTool(BaseTool): name = "multiply_numbers" description = "Useful for multiplying two numbers." args_schema = MultiplyNumbersInput def _run(self, a: float, b: float): return a * b

6. Multi-Agent Systems (CrewAI)

The future of AI is collaborative. Instead of one monolithic agent, we use specialized agents working together.

  • Specialization: A "Researcher" agent finds data, while a "Writer" agent synthesizes it.
  • Collaboration: Agents can delegate tasks to each other, reviewing and refining outputs in a structured workflow.
  • CrewAI: A framework that simplifies multi-agent orchestration, allowing you to define "Crews" with processes and roles.
