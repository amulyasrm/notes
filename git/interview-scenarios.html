1. Daily Essentials & Cleanup

Q1: The "Oops, I missed a file" Committing Scenario
Scenario: You just committed your feature snapshot locally, but suddenly realize you forgot to stage a crucial CSS file. You don't want to create a messy "Forgot CSS" subsequent commit. How do you integrate it?
Answer
git add style.css
git commit --amend --no-edit
Explanation: Using --amend modifies the very last commit. By running --no-edit, you instruct Git to keep the exact same original commit message and just bake the new file into it covertly.
Q2: Un-staging Accidental Files
Scenario: You ran git add . but realized an enormous compiled binary got swept into the Staging Area. How do you remove it safely from Staging without deleting the actual file?
Answer
git restore --staged huge_binary.exe
Explanation: This safely kicks the specified file out of the Index (Staging area), moving it back into the unstaged Working Directory. The physical file remains perfectly unharmed.
Q3: Wiping Untracked Files
Scenario: A build script error generated 50 useless .log files across your directories. Git shows them constantly as Untracked. How do you delete them all permanently via Git?
Answer
git clean -fd
Explanation: clean wipes untracked files. The -f forces the action (Git refuses otherwise out of safety), and -d aggressively removes whole untracked directories, too.
Q4: Finding Historical Files
Scenario: A file was deleted from your project three months ago, but you urgently need its final code structure. How do you look up its history?
Answer
git log --all -- <deleted_file_path>
Explanation: This lists the exact commit history of a file even if it no longer exists inside the current checked-out branch. You can then checkout the file via the last commit hash where it existed.
Q5: Staging "Chunks" of a File
Scenario: You made two changes in utils.js: a brilliant bug fix at the top, and a half-finished broken experiment at the bottom. How do you commit ONLY the bug fix?
Answer
git add -p utils.js
Explanation: The -p stands for "patch". Git will interactively present each isolated "chunk" (hunk) of changes and ask [y/n] if you want to explicitly stage that hunk for the commit.
Q6: The Belated .gitignore Issue
Scenario: Someone committed `.env` secrets into the repo. You add `.env` to `.gitignore`, but Git continues to track changes to it! How do you forcefully stop tracking it?
Answer
git rm --cached .env
git commit -m "Stop tracking environment secrets"
Explanation: Once a file enters Git's history, `.gitignore` can no longer ignore it. You must systematically remove it from Git's internal index cache using `--cached`. This stops tracking it immediately without deleting your local `.env`.
Q7: Rewording the Last Commit
Scenario: You committed a file with the message "Fxed the ui aligment", but realize there is a giant typo. You haven't pushed yet.
Answer
git commit --amend -m "Fixed the UI alignment"
Explanation: Providing a new `-m` dynamically overrides the typo text for the most recent commit. Note: Doing this changes the commit Hash, so never do it if you have already pushed!
Q8: Searching Commit Messages
Scenario: You remember you wrote a commit regarding a "Redis caching" bug fix 3 years ago but don't know where it is. How do you find it?
Answer
git log --grep="Redis caching"
Explanation: This filters the entire commit history precisely searching the commit text messages for the specific phrase.
Q9: Searching Code Changes
Scenario: You need to find the exact commit entirely responsible for introducing a secret generic password string "secret123" into the codebase.
Answer
git log -S "secret123"
Explanation: Referred to as the "Pickaxe" option, `-S` literally searches through the contents/diffs of every file in history to identify exactly which snapshot injected that string.
Q10: Completely discarding all local edits
Scenario: You ran some experiments and broke everything over the past 2 hours locally. You want to absolutely nuke your unsaved edits and reset your code exactly to the last known commit.
Answer
git reset --hard HEAD
git clean -fd
Explanation: This brutal combitation first obliterates all modified tracked files, reverting them back to HEAD, and then follows up by wiping all newly originated untracked files/folders created during your experiment.

2. Branching & Merging Intricacies

Q11: Comparing Branches before Merge
Scenario: You are preparing to merge your feature-alpha branch into main, but you want to see exactly what lines of code differ between the two overall.
Answer
git diff main..feature-alpha
Explanation: The double-dot compares chronologically the tip of `main` directly against `feature-alpha` to show pure code differences you are about to introduce.
Q12: Renaming a Branch
Scenario: You created a branch named `fix-bugg`, and you want to rename it locally to `fix-bug` so you don't look ridiculous.
Answer
git branch -m fix-bugg fix-bug
Explanation: The `-m` (move) command renames the reference branch natively. If you had already pushed the old name, you'd subsequently need to push the new name and explicitly delete the remote old name.
Q13: Restoring specific files from another branch
Scenario: You are on `main`, but you know there is a perfectly functional `config.yml` file sitting over in `branch-beta`. You want to overwrite your `main` copy with that branch's exact version without a complex merge.
Answer
git restore --source branch-beta config.yml
Explanation: Using `restore` with an explicit `--source` tells Git to pull data from another branch (or commit hash) safely into your current working directory.
Q14: Deleting remote branches
Scenario: You deleted a branch locally using `git branch -d feature`, but the branch is still visible on GitHub for the whole team!
Answer
git push origin --delete feature
Explanation: Local branch deletion does not automatically synchronize to remote servers. You must instruct GitHub explicitly via a push operation to erase its held branch reference.
Q15: The Fast-Forward Merge
Scenario: Your interviewer asks: "What constitutes a 'Fast-Forward Merge', and why do some project managers hate it?"
Answer
"A fast-forward occurs when the destination branch hasn't received any new commits. Git simply moves the pointer forward instead of generating a merge commit."
Explanation: PMs sometimes dislike it because the visual "bubble" of the feature branch disappears seamlessly into the linear timeline. To force a merge commit anyway, you use git merge --no-ff feature.
Q16: Safe remote checking
Scenario: Your coworker claims they just pushed some critical logic to `main`. You don't want to immediately pull just in case it triggers horrible merge conflicts holding up your work.
Answer
git fetch
git log --oneline origin/main
Explanation: Fetch securely downloads the remote changes but does NOT instantly merge them into your files. It allows you to preview the cloud status via `origin/main` safely.
Q17: Squashing a Merge Commit completely
Scenario: Your feature branch has 20 horribly named commits (like "wip" or "fix"). You want to merge it into `main`, but want it to look like exactly one clean, massive commit hit `main`.
Answer
git switch main
git merge --squash feature-branch
git commit -m "Add complete feature"
Explanation: `--squash` takes every change embedded across the 20 commits and compacts them into your local staging area safely, allowing you to wrap it all up with one final commit message.
Q18: Setting push default streams
Scenario: You created a new local branch. But when you type `git push`, Git yells at you saying "The current branch has no upstream branch".
Answer
git push -u origin your-branch-name
Explanation: The `-u` (or `--set-upstream`) explicitly links your local branch with a newly created remote branch. Future commits only ever require you to type a simple `git push`.
Q19: Viewing unpublished commits
Scenario: You lost track of how many commits you have made locally that have not yet been successfully pushed to the cloud.
Answer
git log origin/main..main
Explanation: The double dot specifies "Show me everything in the local `main` that doesn't natively exist yet in `origin/main`".
Q20: Pushing an empty folder issue
Scenario: You created 'assets/images' to hold future project images. You commit and push, but your coworker doesn't see the folder. Why?
Answer
Git only tracks FILES, not folders. Create a file inside named .gitkeep.
Explanation: Git fundamentally does not comprehend empty directories. The universally recognized practice is placing a harmless empty file explicitly named `.gitkeep` inside so Git has a reason to track the folder tree.

3. Time Travel & Undoing Safely

Q21: Reverting published work
Scenario: The senior team pushed commit `e5f6g7` to Production. It's crashing user dashboards. Using `git reset` on `main` is heavily frowned upon. What do you do?
Answer
git revert e5f6g7
git push
Explanation: Instead of violently deleting history, `revert` intelligently generates a brand *new* commit specifically designed to inverse/delete all lines of code introduced by the problem commit. Completely safe and auditable!
Q22: Keeping code while dropping commits
Scenario: You created 4 useless commits sequentially, but the actual code works fine! You just want to erase the 4 commits from history while keeping all your raw code completely pristine.
Answer
git reset --soft HEAD~4
Explanation: `--soft` pulls the HEAD pointer natively back 4 spaces precisely, erasing the useless timeline nodes, but dynamically dumps all their raw code smoothly back into your Staging Area.
Q23: The Accidental Merge Reversal
Scenario: You just ran `git merge feature`, dealt with conflicts, and completed the merge. But immediately realized you merged the wrong branch! (You haven't pushed yet).
Answer
git reset --hard ORIG_HEAD
Explanation: When completing sweeping actions like a Merge or Rebase, Git internally caches the previous safe state under a reference variable named `ORIG_HEAD`. Setting your branch there cleanly undoes the local merge entirely.
Q24: The Disastrous 'git reset --hard HEAD~1'
Scenario: You performed a `reset --hard` and lost three critical commits that you literally spent all morning typing. You never pushed them. Are they gone forever?
Answer
git reflog
git reset --hard <surviving-hash>
Explanation: Git rarely actually deletes things immediately. git reflog logs EVERY movement your HEAD makes locally. You can easily find the SHA-hash of the "deleted" commit on the log, and brutally reset your branch directly to it, resurrecting your work from the grave!
Q25: Detached HEAD Nightmare
Scenario: You ran `git checkout e9d8c7` to read historic code. Without noticing, you spent two hours developing new models over it. Git says you are in 'Detached HEAD'. How do you save the project?
Answer
git switch -c rescue-branch
Explanation: Detached HEAD simply asserts you are attached to an isolated commit hash rather than a valid branch pointer. Running `switch -c` literally generates a fresh, permanently-tracked branch pointer exactly upon your location, securing all your modifications.
Q26: Dropping specific staged files
Scenario: You are staring at `git status` indicating you have four modifications waiting inside the Staging Area. You only want to "un-stage" exactly one explicit file named `database.yml`.
Answer
git restore --staged database.yml
Explanation: As covered locally in Essentials, `--staged` shifts tracked files smoothly backward from the Staging Area back into unstaged Working modifications explicitly without nuking edits.
Q27: Stashing Untracked Files
Scenario: You ran `git stash`, but magically, the two totally new files you created are completely unaffected! They didn't go inside the backpack. Why?
Answer
git stash -u
Explanation: By fundamental design, Git explicitly refuses to stash untracked changes because it hasn't technically registered them. By appending `-u` (include-untracked), the backpack forcefully sweeps them inside anyway.
Q28: Examining Stash Items
Scenario: You stashed work last Friday. It is now Monday. You completely forgot what raw code is actually lurking inside that stash stash@{0} before popping it.
Answer
git stash show -p stash@{0}
Explanation: Adding the `-p` (patch) flag forces Git to open the specific stash index and cleanly display the specific internal code diffs precisely without forcing you to `pop` it dynamically.
Q29: Popping vs Applying
Scenario: In a deep interview, they ask: "What is the technical difference between git stash 'pop' versus git stash 'apply'?"
Answer
"Pop applies the data and deletes it from the stash stack. Apply applies the data but retains the backup copy inside the stash array safely."
Explanation: You use `apply` specifically if you want to test the modified stashed code on Branch A, but suspect you might literally want to utilize that identical stash sequentially on Branch B later.
Q30: Undoing "Add" prior to ever committing
Scenario: You initialized a brand new repo (`git init`). You typed `git add .` on massive directories. You want to clear the staging area immediately before executing your very first commit.
Answer
git rm -r --cached .
Explanation: Standard resets might fail critically because a HEAD pointer actively hasn't been born yet technically. Using recursive index cache removal securely blanks the Staging Area dynamically.

4. Advanced Power Tools

Q31: The Scalpel (One specific fix)
Scenario: An experimental physics feature-branch has 30 messy commits. Commit #22 happens to fix a mathematical loop rendering bug. You absolutely must implement that single loop-fix identically into `main` today.
Answer
git checkout main
git cherry-pick <hash-of-commit-22>
Explanation: `cherry-pick` surgically replicates a specifically chosen commit's changes perfectly onto your running branch without sweeping in the 29 surrounding unstable physics commits simultaneously!
Q32: Resolving a Rebase Conflict
Scenario: You are in the middle of executing git rebase main, and suddenly Git stops, throwing glaring red CONFLICT errors regarding a specific controller file.
Answer
1. Fix the file manually in VS Code
2. git add <file>
3. git rebase --continue
Explanation: Just like standard Merges, you resolve markers visually first. Unlike merges, you do not create a new commit. Rebase simply expects you to forcefully stage the fixed file and continue the automated replaying loop immediately.
Q33: Interactive Rebase (Squashing timeline)
Scenario: Your technical lead tells you: "Your feature branch works beautifully, but please logically squash your last 5 commits locally into 1 clean commit before you open a Pull Request."
Answer
git rebase -i HEAD~5
Explanation: Interactive Rebase violently pulls up an editor exhibiting your last 5 commits. You leave the first as 'pick', and manually type 'squash' (or 's') identically adjacent to the other 4. Git fuses them comprehensively into one block.
Q34: The Golden Rule of Rebase
Scenario: Your interviewer demands: "Name the one cardinal rule about using the Rebase functionality within team environments."
Answer
"Never rebase commits that exist physically outside your isolated repository."
Explanation: Because rebasing objectively alters commit cryptography (generates brand new SHA hashes), executing it aggressively on a public shared branch essentially splits reality. Co-workers pulling changes will confront explosive duplicate-commit histories simultaneously.
Q35: Forcing an update post-rebase
Scenario: You pushed your branch yesterday. Today you ran an interactive rebase locally specifically to clean it up. You subsequently type `git push`, but GitHub denies you because your local history doesn't chronologically match the cloud timeline anymore.
Answer
git push --force-with-lease
Explanation: The `-f` or `--force` option overrides GitHub entirely, rewriting remote truth precisely identical to your local truth. The `--force-with-lease` variant is vastly safer as it aborts dynamically if somebody else accidentally triggered a push natively there meanwhile.
Q36: Bisect - The automated detective
Scenario: The product worked perfectly last Friday (Commit A). Today (Commit H), it's totally crashing! There are literally 200 commits injected between A and H. How do you find out exactly which commit introduced the crash in 10 minutes?
Answer
git bisect start
git bisect bad
git bisect good <Commit-A-hash>
Explanation: Git utilizes an intelligent Binary Search algorithm, checking out a commit dynamically halfway through the batch. If you test it and respond "good", Git instantly eliminates 100 commits, dividing the remaining suspects in half perpetually until locating the sole perpetrator!
Q37: Blame - Naming the culprit
Scenario: You are staring at line 125 exactly inside `payments.py` thinking, "Who wrote this horrifying calculation system?"
Answer
git blame payments.py
Explanation: `blame` explicitly prints the precise Author name, temporal timestamp, and specific commit Hash responsible for modifying every solitary line across the specified file!
Q38: Working completely parallel
Scenario: You are heavily modifying backend systems on `branch-A`. The marketing team desperately demands you verify code on `branch-B` currently, but you refuse to stash your huge modifications. Can you view them simultaneously?
Answer
git worktree add ../marketing-fix branch-B
Explanation: `worktree` natively empowers you to spawn a second, totally independent directory immediately adjacent to your project tied identically to the same `.git` core. Highly advanced contextual parallel development.
Q39: Dangling Commits Cleanup
Scenario: You've been coding for three years utilizing Rebase and Hard Resets aggressively. Your hidden garbage commits (dangling data no longer bound to any branch) are taking gigabytes of local storage physically.
Answer
git gc --prune=now
Explanation: Generically acting as Git's "Garbage Collection" mechanism, `gc` intrinsically crawls internally seeking utterly unreachable snapshot fragments explicitly wiping them from the hard-drive sequentially to save massive cache space.
Q40: Stash Conflicts
Scenario: You run `git stash pop` and suddenly receive terrifying 'Merge Conflict' errors from Staging. Why?
Answer
You modified identical lines between stashing and popping.
Explanation: Stashes are technically mini-commits natively tracking specific chronological line patches. Applying them operates exactly identically to cherry-picking or merging. Resolve the conflict directly and the Stash data remains safely integrated.

5. Architecture, Debugging & Web Servers

Q41: The Fork vs Clone Paradigm
Scenario: In Open Source Software, why do you Fork the Linux Kernel instead of simply Cloning the `LinusTorvalds/linux` repository directly?
Answer
Because you lack server write permission natively globally.
Explanation: Cloning merely pulls a local dataset. Forking intrinsically instructs GitHub's cloud platform to duplicate the server-side repo precisely into your own isolated user account globally, granting you explicit Push credentials.
Q42: PR Gatekeeping logic
Scenario: Why does literally every major Tech Company demand Pull Requests rather than permitting direct pushing globally to `main`?
Answer
Safety, Auditing, and Automation sequences.
Explanation: PR's natively intercept code prior to main. This forcibly triggers CI/CD deployment pipelines natively running comprehensive Automated Tests across your snapshot, and simultaneously enforces human peer reviewers critically signing off manually.
Q43: Repositories inside Repositories
Scenario: You are building an enormous enterprise dashboard project. Your company maintains a completely independent Git repo solely tracking universal 'Navigation UI' code they want you to insert organically into your dashboard.
Answer
git submodule add <URL-of-Nav-UI>
Explanation: Submodules elegantly embed distinct Git repositories precisely inside other distinct repositories securely. Your "parent" repo exclusively stores a commit pointer indicating exactly which structural version of the Nav repo you currently rely on.
Q44: Enforcing team rules (Git Hooks)
Scenario: You want to fundamentally block developers natively from ever running `git commit` safely if their raw code totally fails ESLint formatting tests globally.
Answer
Configure a standard pre-commit Git Hook natively globally.
Explanation: Git possesses a hidden `.git/hooks` folder loaded with actionable scripts directly triggering perpetually prior to key actions like pushing or committing. Developers normally utilize package wrappers (like 'Husky') managing these locally.
Q45: Tagging releases securely
Scenario: You just merged version 2.0 correctly into `main`. The DevOps server automatically targets semantic 'tags' fundamentally to deploy. How do you construct an annotated deployment tag?
Answer
git tag -a v2.0 -m "Production Release 2.0"
git push origin v2.0
Explanation: Standard tags merely point globally identically to lightweight branches. 'Annotated' tags (`-a`) compile fully-fledged Git objects embedding comprehensive metadata precisely tracking author origin, timestamps, and deployment messaging.
Q46: Pulling Tags precisely
Scenario: The senior team pushed an urgent tag explicitly representing `v3.1.hotfix` directly to remote. Standard `git pull` randomly didn't fetch it locally securely.
Answer
git fetch --tags
Explanation: Fundamentally reliant on local config states, generic operations occasionally omit transferring independent Tags chronologically. Fetching tags natively secures all newly originated identifiers universally globally.
Q47: Exporting Source safely
Scenario: A business client explicitly requests a raw copy of your complete application code precisely delivered via `.zip`. You inherently cannot hand them a folder natively containing the hidden `.git` database.
Answer
git archive --format zip HEAD > src_code.zip
Explanation: Pushing raw folders locally risks leaking historical architecture and passwords buried within `.git`. Archive intrinsically generates a pristine, fully independent zip explicitly extracting only identical current Working Directory lines globally.
Q48: Understanding Subtree alternatives
Scenario: Submodules are actively annoying your team fundamentally because they constantly demand recursive identical updating explicitly. Are there viable advanced alternatives globally?
Answer
git subtree
Explanation: Subtrees integrate external codebase timelines securely directly into your universal chronological repository completely eliminating independent pointers. It simplifies remote pushing inherently because external code operates essentially identical to local data universally.
Q49: Syncing a stale Fork internally
Scenario: You Forked a massive project directly a year ago locally. Your Fork intrinsically trails the central "Upstream" repository inherently by 300 commits. How do you synchronize exactly globally?
Answer
git remote add upstream <Original-URL>
git fetch upstream
git rebase upstream/main
Explanation: Forks merely communicate identically to 'Origin' structurally. You must explicitly tell Git securely tracking a second remote definitively identified as "Upstream" to pull identical original core lines merging locally successfully.
