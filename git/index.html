What is Version Control?

Without a version control system, you have to constantly store copies of your entire project in various folders. This is slow, doesn't scale, and makes collaboration difficult.

Centralized vs Distributed

  • Centralized: (e.g., Subversion, Microsoft TFS) All team members connect to a central server. If the server goes offline, you cannot save snapshots or collaborate. It has a single point of failure.
  • Distributed: (e.g., Git, Mercurial) Every team member has a full local copy of the project history. You can save snapshots locally even when offline.

Why Git? It is the most popular VCS in the world—free, open source, super fast, and scalable (especially for branching & merging). Over 90% of software projects use Git!

Git Internals: Under the Hood

Before typing commands, you must understand how Git works under the surface. Git doesn't store simple diffs; it stores total snapshots.

Blobs & Trees

A file is represented as a Blob. A folder is a Tree. Git compresses and stores these using cryptographic SHA-1 hashes.

The Commit Object

A commit is purely a snapshot pointer. It holds author metadata, the message, and a reference pointer to its Parent Commit.

HEAD Pointer

HEAD is a literal reference variable that points to the currently checked-out branch. It is your "You Are Here" marker on the map.

Interview prep

Interviewer: How would you explain Git's underlying data model?

You: "Git models data as a Directed Acyclic Graph (DAG) of commits. Unlike SVN which stores file differences over time (deltas), Git stores complete snapshots. Each commit acts as a node pointing to a tree of files, and maintains a pointer back to its parent commit, identified by a cryptographic SHA-1 hash."

Tracking, Staging & Committing

git add file1.txt        # Adds single file
git add *.txt            # Adds all text files

Removing, Moving & Ignoring Files

Removing & Moving

git rm file2.txt      # Removes locally AND stages it
git mv old.js new.js  # Renames file AND stages it

Ignoring Files

Create a .gitignore file to exclude logs or binaries.

*.log              # Ignore all .log files
build/             # Ignore the build directory
⚠️ Common Mistake: Adding a file to .gitignore *after* it was already committed. Git will STILL actively track changes on it because it's already in the cache!

The Fix: You must remove it from Git's index cache first using: git rm --cached file.ext

Branching & Workflows

Branching allows you to diverge from the main line of development, work independently, and later integrate.

git branch -d feature

Safely deletes a branch only if its changes have been merged.

git checkout <branch>

Switches to an existing branch (or commits). Updates your working directory.

git switch <branch>

Modern Alternative! Replaces checkout specifically for switching branches safely.

Real-World Workflows

Git Flow

A structured, older dynamic. Main branches (main, develop) are permanent. Temporary branches (feature/*, hotfix/*) are merged when complete.

Trunk-Based Development

Modern, rapid workflow. Engineers work on very short-lived branches, merging directly into main (the trunk) almost every day to prevent merge nightmares.

The Surgeon's Scalpel: Git Cherry-Pick

What is it? Taking a single specific commit from one branch and applying it to another.

Simple Understanding

  • git merge → brings ALL commits
  • git rebase → reorders ALL commits
  • git cherry-pick → picks ONLY ONE (or a few) commits
git cherry-pick <commit-hash>

Visual Idea

feature branch:  A --- B --- C (bug fix) --- D
main branch:     A --- B

After cherry-pick (on main):
main branch:     A --- B --- C (new hash)

Real Scenario

Situation: You fixed a massive bug inside a feature branch containing dozens of experimental additions. Your boss desperately needs that single fix in main right now. Don't merge the half-finished feature yet!

git switch main
git cherry-pick abc1234

Only the isolated bug fix safely copies over!

⚠️ Important Truths (Don't Ignore):
  • Creates a NEW exact-copy commit: The snapshot is the same, but the SHA hash generated is entirely new.
  • Conflicts happen: If it hits a conflict, resolve it, array the changes, then explicitly run git cherry-pick --continue.
  • Don't Overuse it! If you continually duplicate commits across branches randomly, your graph history will become unreadable.
✅ When to use ❌ When NOT to use
Hotfixes: A sudden fix needed across multiple release branches. Large features transfers: Moving massive, interconnected workflows (Use Merge).
Selective sharing: Taking one isolated feature without pulling the dirty branch. Complex shared history: Trying to cherry-pick highly dependent commits causes severe conflict hell.

Merge vs. Rebase

When bringing code from one branch to another, you have two radically different choices.

Merge

Integrates code non-destructively by creating a new Merge Commit.

  • Pro: Keeps true, chronological history.
  • Con: History can look chaotic if there are many branches.
git merge feature-branch

Rebase

Plucks your commits, temporarily removes them, fast-forwards the base, and reapplies them. Rewrites history.

  • Pro: Clean, perfectly linear history.
  • Con: Dangerous. Overwrites commit SHAs.
git rebase main
⚠️ THE GOLDEN RULE OF REBASING: Never rebase commits that exist on a public, shared branch.

Rebase locally on your private branches to keep them clean. But if you rebase `main`, you rewrite public history meant for everyone else, causing horrific duplicate commits when they pull.

Handling Merge Conflicts

A conflict happens when Git cannot automatically merge because the same line of code was edited in two different commits.

What literal Git conflict markers look like in code:

function calculateNetTotal() {
<<<<<<< HEAD (Current Branch)
    return basePrice + tax + shipping;
======= (Divider)
    return basePrice + stateTax + fee;
>>>>>>> feature-branch (Incoming Branch)
}

How to fix it

  1. Open the file in an editor like VS Code (it will highlight these blocks).
  2. Choose which block of code to keep (Current, Incoming, or manually mesh them together).
  3. Delete the markers (<<<<, ====, >>>>).
  4. Run git add <file> and git commit --continue to complete.

Git Stash (The Quick Backpack)

Useful when deeply modifying files, but you must switch contexts immediately without committing half-finished code.

  • git stash - Hides your temporary changes into a stack. Work directory becomes clean.
  • git stash pop - Restores the topmost stashed files back into your working directory.
  • git stash list - Reviews the currently stashed list array.

Restoring & Undoing Operations

Git allows you to safely (and unsafely) step backward when things break!

git restore

git restore file.js overrides local un-staged edits with staging copy. Safer alternative to checkout file.

git clean -fd

Permanently force-deletes newly generated untracked files and directories.

git commit --amend

Missed a file? Typo in message? Modify the last commit without adding a new messy history log.

Deep Undo: Reset vs Revert

git reset

Moves the HEAD pointer backwards. Actually erases commits from the history timeline altogether. (Dangerous)

git revert

Creates a brand new commit that inverses the changes of a previous commit. (Safe for teams!)

Reset Mode Where do my uncommitted edits go? Safety Level
git reset --soft HEAD~1 Files shift seamlessly into the Staging Area. Green (Safe)
git reset --mixed HEAD~1 Files fall back into the Working Directory (Unstaged). Default. Yellow (Caution)
git reset --hard HEAD~1 Files and edits are DELETED instantly forever. Red (Danger!)
