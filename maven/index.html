PART 1: The Core Fundamentals

Fundamental

1. The Maven Project Structure

Maven revolutionized Java by explicitly enforcing "Convention over Configuration." Before Maven, everybody structured files differently. Maven demanded one universal pattern. If you break this structure visually, your project literally will not build.

my-app/
src/
main/
java/ (Your raw App Code)
resources/ (Images, properties, YAMLs)
test/
java/ (Your JUnit tests)
resources/ (Test-only configs)
target/ (Auto-generated compiled output)
pom.xml (The absolute core config)

Why does this exact structure exist?

Because Maven's inherent source code natively looks inside `src/main/java` by default. If you place a file freely in the root directory like a Python script, Maven literally ignores it entirely during the `compile` phase!

Fundamental

2. The Repository Triad

You already know Maven downloads .jar files. But it actively checks three completely distinct locations structurally.

Repository Type Location Purpose
1. Local Repo ~/.m2/repository (Hidden on your PC) Maven always checks here first aggressively to save internet bandwidth. It acts as your local cache.
2. Central Repo Internet (Global Maven Server) The world's open-source library. If a JAR isn't physically in your `.m2` folder, Maven downloads it from here and physically caches it locally.
3. Remote Repo Your Company's Private Server (Nexus/Artifactory) Used entirely in enterprise! Your company builds classified tools and actively hosts them internally safely away from the public Central Repo.

Critical Concept

3. The Grand Illusion: Maven Plugins

The Brutal Truth: Maven itself does absolutely nothing!

Maven is fundamentally just an empty shell. It intrinsically cannot compile code. It cannot run tests natively. Everything Maven achieves is actually mechanically executed by Plugins.

When you type `mvn compile`, Maven natively triggers the maven-compiler-plugin. When you type `mvn test`, it triggers the maven-surefire-plugin!

Customizing Plugins in pom.xml

<build>
    <plugins>
        <plugin>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.8.1</version>
            <configuration>
                <!-- By default Maven uses Java 1.5 (ancient). We force Java 17 like this: -->
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>
    </plugins>
</build>

Without explicitly configuring plugins, your build cycle operates blindly on outdated default specs.

Interview Lifesaver

4. Dependency Scopes

Adding a dependency isn't enough. You must precisely tell Maven exactly WHEN that dependency is structurally allowed to execute. Interviewers ALWAYS ask this.

<dependency>
    ...
    <scope>test</scope> <!-- Defining the boundary limits -->
</dependency>
Scope Meaning & Usage
compile (Default) Included actively in compiling, testing, and running physically packaged into your final app securely.
test Only explicitly injected during the `mvn test` phase (e.g., JUnit or Mockito). Systematically blocked from entering the final production `jar/war` file to save space.
provided (Crucial) Required locally to compile, but the physical Web Server (Tomcat) will inherently provide the Jar exclusively at runtime. If you fail to use this, you'll cause massive duplicated-library server crashes.
runtime Not inherently needed maliciously to explicitly compile code, but absolutely demanded mechanically during execution natively (e.g., JDBC Database Drivers).
